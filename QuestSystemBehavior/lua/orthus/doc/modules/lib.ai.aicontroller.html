Module lib.ai.aicontroller

This module allows to create AI player and armies.

There is no limit to armies per player nor troops per army.

Armies have their script behavior defined by attack and defence positions on the map. An AI player can automatically employ armies. Also armies can be hidden from the AI to control them in a own job.

If you are an beginner you should stick to the options the quest behaviors give you. They are enough in most of the cases.

Required modules:

  • qsb.oop
  • qsb.quest.questsync
  • qsb.quest.questtools
  • qsb.ai.aitrooprecruiter
  • qsb.ai.aitroopspawner
  • qsb.ai.aiarmy

ArmySetController (_Army, _Function, ...)
Sets an manual controller for an army.

When the controller is set the army is automatically hidden from the AI. If the controller is invalidated by passing nil the army is returned to the AI.

If the controller is add successfully the ID of the created job is returned. Otherwise nil is returned.

The controller is called each second and the ID of the army is passed. To access the army data index AiArmyList with the army ID. Inside your function you only need to add behavior to the army.

Note: This feature is considered advanced level. In most cases the normal AI controller is sufficient to do the job. Use this if you need multiple bases for one AI which will not work with the generic controller.

Note: See documentation of AiArmy for futher information.

Parameters:

  • _Army Name or ID of army
  • _Function function Controller function
  • ... Controller argument list

Returns:

    number Job ID of controller

Usage:

    -- Function reference
ArmySetController("SomeArmy", MyControllerFunction);
-- Inline function
ArmySetController("SomeArmy", function(_ArmyID)
    -- Do something here
end);
ArmySetExemtFromAttack (_Army, _Flag)
Removes the army from candidates for attack operations or rejoins it. When an army is removed or returned, all objectives of this army are nullified.

Armies that won't go on patrol will remain on their home position when not ordered to patrol.

Note: This will only affect armies with default controller. If a custom controller is used, this won't have an effect.

Parameters:

  • _Army Name or ID of army
  • _Flag boolean Army can't attack

Usage:

    ArmySetExemtFromAttack("SomeArmy", true);

ArmySetHiddenFromAI (_Army, _Flag)
Hides an army from their AI or returns them back. Hidden armies will not be considered when the AI selects targets to attack or to defend. When an army is hidden or returned, all objectives of this army are nullified.

Note: Armies will be automatically hidden when a custom controller is added to them. Use this only if you want the army to be stationary without receiving any orders. (Bandits camp ect.)

CreateAIPlayerArmy (_ArmyName, _PlayerID, _Strength, _Position, _Area)
Initalizes an army that is recruited by the AI player. Armies can also be created with the behavior interface. This is a simple type of army that can be configured by placing and naming script entities. The army name must be unique for the player!

The AI player must be initalized first!

Use script entities named with PlayerX_AttackTargetY to define positions that will be attacked by the AI. Replace X with the player ID and Y with a unique number starting by 1.

You can also use entities named with PlayerX_PatrolPointY to define positions were the AI will patrol. Also replace X with the player ID and Y with a unique number starting by 1.

Note: The AI will decide which targets an army is send to. There isn't a direct connection for one army and one target. If you don't want the AI to controll the army it must be hidden from the AI. In that case you have to provide the army with targets.

Parameters:

  • _ArmyName string Army identifier
  • _PlayerID number Owner of army
  • _Strength number Strength of army [1|8]
  • _Position string Home Position of army
  • _Area number Action range of the army

Returns:

    table Army

Usage:

    CreateAIPlayerArmy("Foo", 2, 8, "armyPos1", 5000);
CreateAIPlayerSpawnArmy (_ArmyName, _PlayerID, _Strength, _Position, _Spawner, _Area, _RespawnTime, ...)
Initalizes an army that is spawned until a generator entity is destroyed. Armies can also be created with the behavior interface. This is a simple type of army that can be configured by placing and naming script entities. The army name must be unique for the player!

The AI player must be initalized first!

Define a list of troops that are sparned in this order. Troops will be spawrned as long as the generator exists.

Use script entities named with PlayerX_AttackTargetY to define positions that will be attacked by the AI. Replace X with the player ID and Y with a unique number starting by 1.

You can also use entities named with PlayerX_PatrolPointY to define positions were the AI will patrol. Also replace X with the player ID and Y with a unique number starting by 1.

Note: The AI will decide to which targets an army is send to. There isn't a direct connection for one army and one target. If you don't want the AI to controll the army it must be hidden from the AI. In that case you have to provide the army with targets.

Parameters:

  • _ArmyName string Army identifier
  • _PlayerID number Owner of army.
  • _Strength number Strength of army [1|8]
  • _Position string Home Position of army
  • _Spawner string Name of generator
  • _Area number Action range of the army
  • _RespawnTime number Time till troops are refreshed
  • ... table List of types to spawn

Returns:

    table Army

Usage:

    CreateAIPlayerSpawnArmy(
    "Bar", 2, 8, "armyPos1", "lifethread", 5000, 2*60,
    {Entities.PU_LeaderSword2, 3},
    {Entities.PU_LeaderBow2, 3},
    {Entities.PV_Cannon2, 0}
);
